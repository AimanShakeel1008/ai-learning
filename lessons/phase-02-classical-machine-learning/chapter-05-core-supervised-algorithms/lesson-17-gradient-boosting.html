Gradient Boosting

Many small models, each one fixing the mistakes of the ones before it.

Files created: lesson-notebooks/phase-02-classical-machine-learning/chapter-05-core-supervised-algorithms/lesson-17-gradient-boosting.ipynb

Prerequisites: Lesson 9 (gradient descent), Lesson 13 (overfitting), Lesson 14 (residuals, squared error, learning rate), Lesson 16 (decision trees, greedy splits, random forests)

1Why this matters

In the last lesson we built a random forest. A forest builds many trees separately, and the trees vote. At the end of that lesson we promised there is a second, stronger way to build a team of trees. This lesson keeps that promise.

The new method is called gradient boosting. It builds its trees one after another, in order. Each new tree has exactly one job: fix the mistakes that all the trees before it are still making.

That one small change turns out to be very powerful. It is the idea behind famous tools like XGBoost and LightGBM. For more than ten years, these tools have won most machine-learning competitions on tabular data — data that fits in a table, like rows of tickets, orders, or customers. When a data scientist at a real company gets a table of business data and wants the strongest classical model, "try gradient boosting" is almost a reflex. If you only ever learn two classical model families really well, pick these two: linear models (Lessons 14–15) and boosted trees (this lesson).

There is also a nice bonus waiting at the end. The word "gradient" in the name is not an accident. This lesson closes a loop that started in Lesson 9: boosting is gradient descent — just taken one small tree at a time. And for our support assistant, this lesson is a preview of how its predictions (resolution time, urgency, category) would be built with industrial-strength tools.

2Real-world analogy

Editing an essay in passes.

You write a first draft of an essay in five minutes. It is rough — but it exists. An editor reads it and marks, in red pen, exactly what is still wrong: "this paragraph is too vague," "this claim has no support." You then make one revision pass. In that pass, you fix only what the red pen marked. Next, the editor reads the new version and marks what is still wrong. The list is shorter now. Another pass fixes those. After several passes, the essay is sharp — even though no single pass was brilliant on its own.

Here is how each piece maps to boosting:

  • The rough first draft = the starting model. It is just one boring average, predicted for everyone.
  • The red-pen marks = the residuals. They are a precise list of what the current model still gets wrong, ticket by ticket.
  • One revision pass = one small tree. It is trained on the red-pen marks — not on the original assignment.
  • Each pass reads the latest version = each tree is trained on the errors left over after all the previous trees. The work happens in order: pass 3 cannot happen before pass 2.
  • A careful editor who only changes 30% of what they could = the learning rate. Apply each correction only partly, so no single pass pushes its opinion too hard.
  • Knowing when to stop editing = the number of rounds. If you keep revising an essay that is already good, you start ruining good sentences. That is overfitting.

Compare this with Lesson 16's forest. The forest was five people writing five separate essays and voting. Boosting is one essay, improved by many passes.

3The concept explained

3.1 The starting point: the most boring model alive

Boosting starts with a "model" that ignores its input completely. It always predicts the average of the training answers. If our 8 past tickets took 24.5 minutes on average, it predicts 24.5 minutes — for every ticket, without looking at anything.

Why the average, and not some other number? Because we score models with squared error (Lessons 9 and 14). If you are forced to predict one single number for everyone, the average is the number with the smallest total squared error. It is the best you can do without looking at the input. Everything that comes after this is about looking.

3.2 Residuals: the list of what is still wrong

A residual (we met it in Lesson 14) is a simple subtraction: actual answer minus predicted answer. There is one residual per training example. If a ticket actually took 8 minutes and we predicted 24.5, the residual is 8 − 24.5 = −16.5. That means we guessed 16.5 minutes too high. A positive residual means we guessed too low. A negative one means we guessed too high.

Here is the key way to think about residuals: they are not just a score. They are a to-do list, one item per ticket. "On this ticket, move your prediction down by 16.5. On that one, move it up by 21.5." The central trick of boosting is this: use that to-do list as the new training target.

3.3 The weak learner: a decision stump

The corrections are made by tiny trees — tiny on purpose. A decision stump is a decision tree that is allowed exactly one question: one split, two leaves. Our stumps ask "is the ticket length ≤ some threshold?". Each side of the split answers with a single number.

Why use a weak piece on purpose? Because the plan is many small corrections. Each stump fixes a little bit of what is wrong. Whatever it misses stays in the residuals, so the next stump gets a chance at it. Now imagine the opposite: one strong model that fixes everything in one go. It would also learn all the noise in one go — that is Lesson 13's overfitting. Weak pieces, added slowly, are easy to keep under control. The boosting literature literally calls them weak learners.

How do we fit a stump? It is Lesson 16's greedy split search, with two small swaps because we now predict numbers instead of labels:

  1. The candidate thresholds are the midpoints between neighbouring lengths (15, 25, …, 75).
  2. Each threshold splits the tickets into a left pile (≤) and a right pile (>).
  3. Each side's answer is the average of its pile — the best single number, for the same reason as in 3.1.
  4. Each threshold's score is the squared error left over. The lowest score wins.

In Lesson 16, a leaf answered with the majority label, and splits were scored with Gini. Here, a leaf answers with the average, and splits are scored with squared error. Everything else is the same machine.

3.4 The boosting loop, worked with real numbers

Here is our notebook data: ticket lengths in words, and the minutes each ticket took to resolve. Notice the jumps — this data is not a straight line:

length:  10  20  30  40  50  60  70  80
minutes:  8  10  14  16  28  30  44  46

Round 0. The average is 196 ÷ 8 = 24.5. The residuals are: −16.5, −14.5, −10.5, −8.5, +3.5, +5.5, +19.5, +21.5. The total squared error is 1550.0.

Round 1. Fit a stump to the residuals. We try every threshold, and "length ≤ 45?" wins easily. Why? Because the biggest jump in the data sits between 40 and 50 words. On the left of that line, the residuals average −12.5. On the right, they average +12.5. Now add the correction:

new prediction = old prediction + stump's correction
left of 45:  24.5 + (−12.5) = 12.0 minutes
right of 45: 24.5 + (+12.5) = 37.0 minutes

The total squared error drops from 1550 to 300. One yes/no question removed 81% of the error.

Round 2. Compute the residuals again — but against the NEW predictions: −4, −2, +2, +4, −9, −7, +7, +9. The biggest remaining problem is on the right side: the 70–80-word tickets are still guessed 7–9 minutes too low. The best stump is "length ≤ 65?", with corrections −2.67 on the left and +8.0 on the right. The error falls from 300 to 129.3.

Round 3. The best stump is "length ≤ 45?" again — but this time with small values (+2.67 / −2.67). It is fine-tuning a region it already corrected once. The error falls from 129.3 to 72.4.

So each round does the same four things: (1) compute the residuals, (2) fit a stump to them, (3) add the stump's corrections to the model, (4) repeat. The final model is a recipe:

prediction(ticket) = 24.5                      <- the average
                   + stump1(ticket)            <- ±12.5 at the 45-word line
                   + stump2(ticket)            <- fixes the right side
                   + stump3(ticket)            <- fine-tuning
                   + ...

Notice why this is sequential (one-after-another). Stump 2's training target is the round-2 residuals — and those did not exist until stump 1 was built and applied. No tree can be trained before the trees that come before it. Forest trees are like jurors voting at the same time. Boosting trees are like runners in a relay race.

3.5 The learning rate: take smaller steps on purpose

Real boosting rarely applies a stump's full correction. Instead, it multiplies every correction by a small number first. That number is the learning rate, also called shrinkage. Typical values are 0.3, 0.1, or smaller:

new prediction = old prediction + learning_rate × correction

Why would you weaken your own corrections on purpose? Remember: each correction was measured on the training tickets only. So it is partly real pattern and partly noise (Lesson 13). A small step means no single stump can push its full opinion — noise included — into the model. And the real pattern is not lost. Whatever a small step leaves unfixed is still sitting in the residuals. So later rounds see it again, and keep correcting it. Real structure gets confirmed round after round. One-off noise does not. The price is that you need more rounds — which is why production boosted models use hundreds of small trees.

This is the same careful-versus-fast dial as the learning rate in gradient descent (Lessons 9 and 14). That is not a coincidence, as the next section shows.

3.6 Why "gradient" boosting

From Lesson 9: the gradient points in the steepest-uphill direction on the error surface, and gradient descent means "step a little downhill, then repeat." Now look at what boosting does. With squared error, ask: for one prediction, which direction reduces its error fastest? The answer is exactly its residual. "You predicted 12, the answer is 16" means "move up by 4." So when boosting fits a stump to the residuals and adds a small multiple of it, it is taking a gradient-descent step. The only difference is the kind of step: instead of nudging a weight and a bias, the step is a whole small tree. Boosting is gradient descent where each step is a tree. (For error measures other than squared error, libraries fit each tree to a generalized version of the residual — the idea is exactly the same.)

3.7 Where boosting overfits — and its knobs

In a forest, extra trees are just extra independent opinions going into an average. So for a forest, more trees is roughly always safe. Boosting is different. Every extra boosting tree keeps chasing whatever error is left. And once the real pattern has been captured, the only thing left to chase is noise. So boosting can push its training error all the way to zero and appear to keep "improving" — while quietly getting worse on new data. The knobs that control this:

  • the number of rounds — more trees means more capacity to memorize,
  • the learning rate — smaller means more careful,
  • the depth of each tree — deeper means stronger pieces, which also means more capacity.

All of these are tuned with a validation set (Lesson 12). The standard trick is early stopping: watch the validation error after every round, and stop adding trees when it stops improving.

3.8 The real libraries

Nobody writes this by hand in production. XGBoost and LightGBM are the famous stand-alone tools, and scikit-learn ships a histogram gradient-boosting model too. They all run exactly the loop above, plus some upgrades: trees a few levels deep instead of stumps, hundreds of rounds at small learning rates, random subsets of rows and columns per tree (tricks borrowed from the forest), penalties that keep the trees simple, early stopping, and serious speed engineering.

Library names and argument details change over time — treat any specific API in this lesson as something to verify against the official docs.

6Applied to our assistant

No project change this lesson — on purpose. The assistant's resolution-time data (Lesson 14) was built to sit on a straight line, so its linear model already fits it well. Swapping that good model for a from-scratch stump booster would make the project worse, not better.

Gradient boosting shines on bigger, messier tables with many features. And doing it properly means using a real library (scikit-learn, XGBoost, and friends), which arrive later in the course. 7Common mistakes and gotchas

  • Training each new tree on the answers instead of the residuals. If you do that, every tree learns the same thing, and adding them together makes the predictions absurdly large. The target for round n is what is still wrong after rounds 1 to n−1. That detail is the entire algorithm.
  • Assuming "more trees" is safe, like in a forest. In a forest, extra trees just make the average more stable. In boosting, extra rounds add memorizing power, and will eventually memorize noise. Boosting needs early stopping or a tuned round count. A forest mostly does not.
  • Turning the learning rate up to train faster. Big steps let individual trees stamp their noise into the model. The reliable recipe is the opposite: a small learning rate, many rounds, and early stopping.
  • Using strong (deep) trees as the weak learner. A depth-8 tree grabs a big chunk of pattern-plus-noise in one bite, and boosting then piles those big bites on top of each other. Real setups keep trees shallow (commonly about 3–8 levels; our notebook uses depth 1) so the loop stays in control, not the pieces.
  • Expecting smooth predictions. A sum of trees is a staircase. All inputs between two thresholds get the exact same prediction, and beyond the outermost threshold the prediction goes flat. This is the same can't-extrapolate blindness as Lesson 16's trees.

8When to use this, and tradeoffs

Reach for gradient boosting when your data is a table of mixed features and you want the strongest classical accuracy available. It is the default "serious model" for tabular business problems — churn, fraud, pricing, or our ticket predictions at scale — and once tuned, it usually beats a random forest by a little. Everything trees gave you carries over: no feature scaling needed, and mixed feature types are welcome.

The costs. It must be tuned (rounds, learning rate, depth) — a forest is much more forgiving straight out of the box. It trains one tree after another, so it is harder to speed up by throwing more machines at it. A sum of 300 trees is not something a human can read, the way Lesson 16's single printed tree was. And because it chases residuals, it will chase mislabeled examples too — noisy labels hurt boosting more than they hurt a forest. On tiny datasets (like our 8 tickets!) it happily overfits. For text-heavy meaning problems, later phases' neural models take over. An honest default: try a random forest first as a robust baseline, then reach for boosting when you want more accuracy and are willing to tune.

9Key takeaways

  • Gradient boosting starts from one boring guess (the average), then repeatedly fits a small weak tree to the current residuals and adds a scaled-down copy of its corrections to the model — so the final model is the average plus a long sum of small corrections.
  • Each tree is trained on the errors left over after all the previous trees, which forces boosting to build its trees one after another — the opposite of a random forest's independent trees voting in parallel.
  • For squared error, each prediction's residual is exactly the direction that reduces its error fastest, so boosting is gradient descent in which every step is a whole small tree — that is what the "gradient" in the name means.
  • The learning rate shrinks every correction so no single tree can stamp its noise onto the model, at the price of needing many more rounds; the number of rounds, the learning rate, and the tree depth are boosting's overfitting knobs, tuned with a validation set and early stopping.
  • In practice you use a library — XGBoost, LightGBM, or scikit-learn's histogram gradient boosting (verify current APIs against official docs) — and this family is usually the strongest classical choice on tabular data.

1. Round 0. The average is (10 + 12 + 30 + 32) ÷ 4 = 21. The residuals are actual − 21: −11, −9, +9, +11. The total squared error is 121 + 81 + 81 + 121 = 404.

2. Round 1. The one big jump in this data sits between 20 and 30 words, so "length ≤ 25?" should win. Checking all three thresholds:

  • Threshold 15: left pile = {−11}, average −11, error 0. Right pile = {−9, +9, +11}, average 3.67, leftover error (−12.67)² + (5.33)² + (7.33)² ≈ 242.7. Total ≈ 242.7.
  • Threshold 25: left pile = {−11, −9}, average −10, error 1 + 1 = 2. Right pile = {+9, +11}, average +10, error 1 + 1 = 2. Total = 4.
  • Threshold 35: the mirror image of threshold 15, so also ≈ 242.7.

The winner: "length ≤ 25?" with corrections −10 (yes) and +10 (no).

3. Learning rate 1.0. The predictions become 21 − 10 = 11 for the two short tickets, and 21 + 10 = 31 for the two long ones. The residuals are −1, +1, −1, +1. The total squared error is 4, down from 404. One question removed 99% of the error, because this toy data is one clean step.

4. Learning rate 0.5. The corrections shrink to ±5. The predictions become 16, 16, 26, 26. The residuals are −6, −4, +4, +6. The total squared error is 36 + 16 + 16 + 36 = 104. Round 2 would likely pick threshold 25 again. Why? Half of the jump is still unfixed, so the biggest remaining pattern is still "short side too high, long side too low." This is the learning-rate trade in miniature: each pass fixes less, but the unfixed part stays on the to-do list (the residuals) and gets corrected again in later passes.
